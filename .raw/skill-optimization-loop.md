## Post: Building a skill optimization loop

This post shows how to create a loop with automated feedback that an agent can run to optimize its own Skills. It uses an automated grader with computer use to assess how well a Skill is performing, and then iteratively improves the Skill.

You can apply this technique to create improvement loops any time a Skill has clear validation criteria.

### The setup

Example skill: `/replatform-site` — migrates websites from WYSIWYG platforms (Squarespace, Webflow, Wix, WordPress, Shopify) to static, source-controlled code deployable to Vercel/Netlify/Cloudflare Pages.

Initial run on talkingslop.ai: got pretty far, but had one obvious visual defect (missing icons for dropdown toggles).

### Two loops you might implement

1. **Inner loop** — a loop to make sure this specific replatforming worked
2. **Outer loop** — a loop to make the replatforming Skill itself more likely to work better next time

This post focuses on the **outer loop** — the skill improvement loop.

### The observer skill pattern

Set up the loop through another Skill: an "observer" skill (`/replatforming-observer`) that grades the quality of the "inner" Skill.

**Observer skill inputs:**
- N sites to replatform
- Inner skill path (default: `../replatform-site/SKILL.md`)
- Framework/host settings
- Viewports: desktop 1440x900 and mobile 390x844
- Maximum DIF iterations: default 3
- Workspace path

**Observer skill pipeline:**
1. Calls `/replatform-site` on each input site
2. Builds generated sites locally
3. Examines them for behavioral and visual differences using **computer use + browser use**
4. Tracks token counts and attempts to optimize cost while maintaining quality
5. Synthesizes results using a SOTA model
6. Looks for failure patterns and opportunities to improve
7. Creates a **diff** to update the inner Skill

Because Skills are just files, any coding agent (like Warp) can do this analysis and submit a PR to update the Skill.

### Structured data results

The observer Skill uses structured data results so it can be as intelligent as possible when suggesting fixes to the inner Skill. It's fairly sophisticated, but the general concept is simple: run the inner Skill, record its failures, make a diff to improve it, repeat.

### Orchestration requirement

To run this loop you need a platform that supports orchestration of multiple agents along with computer use. The example uses Oz (built into Warp, supports computer use across multiple SOTA models).

### Exit criteria

The observer has exit criteria baked into it for when to stop looping — so you don't burn tokens optimizing forever. If you want to tune this Skill on a significant corpus for wider use, scale the number of input sites and keep iterating until the diffs generated by the observer become less meaningful.

### Key insight

This system isn't perfect — it's susceptible to finding local maxima — but it's pretty handy as a simple way of making sure your Skills perform well.

**The inner vs. outer loop distinction:**
- Inner loop: make sure this specific run worked
- Outer loop: make the Skill itself better for future runs

Skills are just files → any coding agent can modify them → skill improvement is just a code change.

## Replatformer README (github.com/warpdotdev-demos/replatformer)

### Purpose

Two goals:
1. **Demonstrate re-platforming** — end-to-end workflow moving a hosted site into deployable static code
2. **Demonstrate an outer-loop skill-improvement loop** — the skill itself is tuned by an automated outer loop that uses computer use to run the skill against example sites, evaluate the output for correctness and token efficiency, and feed the results back into the skill definition

### What the inner skill does

Given a hosted website:
1. **Inventories** the source site — pages, assets, navigation, content, metadata
2. **Extracts** content and structure into a static site framework (Next.js/Astro by default)
3. **Normalizes** assets into the repo (no longer depends on host's CDN)
4. **Re-points** forms, search, and other dynamic features to serverless equivalents
5. **Generates** a deploy-ready project with static host config (e.g., `vercel.json`)
6. **Verifies** the migrated site renders equivalently before cutover

### Project layout

```
replatformer/
├── README.md
└── .agents/skills/
    ├── replatform-site/
    │   ├── SKILL.md              # Inner replatforming skill
    │   └── references/
    │       ├── providers.md      # Provider gathering & feature re-pointing
    │       └── frameworks.md     # Framework scaffolding & deploy config
    ├── replatforming-observer/
    │   ├── SKILL.md              # Visual/efficiency outer improvement loop
    │   ├── references/           # Artifact schema and failure taxonomy
    │   ├── scripts/              # Deterministic metrics aggregation
    │   └── evals/                # Representative observer evaluations
    └── oz-orchestrated-replatforming/
        ├── SKILL.md              # Parallel Oz cloud execution driver
        ├── references/           # Cloud result and handoff schema
        ├── scripts/              # Deterministic result validation
        └── evals/                # Representative orchestration evaluations
```

### Key structural decisions

- Main workflow in SKILL.md; provider- and framework-specific detail in `references/` to keep core instructions concise (progressive disclosure)
- The base skill commit is recorded so improvement loops can diff against it as a baseline
- Observer runs controlled baseline/candidate migrations, compares source and generated sites with computer vision, measures quality and efficiency, feeds evidence-backed differential improvements into the base skill
